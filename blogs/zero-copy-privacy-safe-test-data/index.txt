14:T59e8,<p><em>A generalized writeup of a production system I built — the patterns, not the proprietary details.</em></p>
<p>A fleet of devices in the field each produces a "session": a folder of files containing manifests, control records, metadata JSON, and a stack of large binary recording chunks. A typical session runs about 20 GB, and roughly 17 GB of that is binary recording data. Those recordings are deterministic — they're byte-identical across every test variant of a session. Everything else — a few hundred megabytes of metadata — is what carries the sensitive identifiers a test pipeline needs scrubbed.</p>
<p>QA needed privacy-safe variants of these sessions at volume. The existing flow was a manual ritual: download a session from object storage to a local runner, hand-edit identifiers in JSON files to produce a "test variant," re-upload the modified session, run the test. They did this for a hundred-odd sessions, several times a day, on runners with a fixed local disk budget.</p>
<p>Three things were structurally broken.</p>
<ul>
<li><strong>The runners couldn't hold the data.</strong> A hundred-plus sessions at ~20 GB each blew past local disk constantly.</li>
<li><strong>Egress was bleeding.</strong> Each day's runs re-downloaded terabytes — almost all of it binary recording bytes the test would never touch.</li>
<li><strong>Scrubbing was manual and unprovable.</strong> A human opened JSON in an editor and replaced identifiers by hand. No audit trail, no validation, no way to prove a generated dataset was actually clean.</li>
</ul>
<h2>Why the obvious approach fails</h2>
<p>The obvious fix is "make the copy pipeline faster" — parallelize the download, compress in transit, cache hot sessions. But every version of that still moves the 86% of bytes that are identical between the source and its variant. You can make copying cheaper; you can't make it free, and copying is the whole cost.</p>
<p>So the real question is not "how do we copy faster" but "why are we copying at all?" If the overwhelming majority of every variant is bit-for-bit the same as its source, the right move is to <em>not copy those bytes</em> — to point at the originals and materialize only the difference. That requires a substrate where a "version" of a dataset is a pointer, not a duplicate. A versioned data lake — lakeFS, in this case, backed by GCS object storage with a zero-copy staging primitive and an in-process hook engine — is exactly that substrate.</p>
<h2>High-level architecture</h2>
<p>The control plane is always on and cheap; the data plane scales from zero. KEDA bridges them by scaling workers off queue depth.</p>
<figure class="diagram">
  <img class="diagram-light" src="/diagrams/zero-copy-architecture-light.svg" alt="Zero-copy test-data flow: a variant is a branch, validated then merged; on merge an in-lake hook fans out per-file messages to a consumer that server-side-copies to the destination bucket." />
  <img class="diagram-dark" src="/diagrams/zero-copy-architecture-dark.svg" alt="Zero-copy test-data flow: a variant is a branch, validated then merged; on merge an in-lake hook fans out per-file messages to a consumer that server-side-copies to the destination bucket." />

<p>Two logical stores do the work: a <strong>catalog repo</strong> (every uploaded session, imported zero-copy when it's tagged) and a short-lived <strong>run repo</strong> per test run (deleted when the run finishes). Only two human actions exist in the whole flow: a session lands in the bucket and the tagger picks it up automatically; a user clicks "Generate" and everything from run-repo creation through delivery happens with no further input.</p>
<h2>Pattern 1: the branch is the variant</h2>
<p>A test variant is a branch, not a copy, not an export, not a "rendered" session. There is no materialization step. Generation is: create a branch, walk the source files, replace identifiers in the ~2% of files that contain them, rename the files whose names embed identifiers, commit.</p>
<p>The primitive that makes this work is zero-copy staging. The orchestrator registers a metadata pointer that references the physical object already managed by the catalog repo — no bytes move:</p>
<pre><code class="hljs language-python">client.stage_object(
    repository=run_repo,
    branch=source_branch,
    path=dst_path,
    mode=<span class="hljs-string">"zero-copy"</span>,
    physical_address=src_physical_address,
)
</code></pre>
<p>A multi-gigabyte session stages into the run repo in seconds.</p>
<p>One structural bug here is worth dwelling on. An early orchestrator branched every variant from <code>main</code>. Runs that combined two sessions started leaking identifiers across them — a variant of session A would occasionally pick up a token that originated in session B's metadata. The fix wasn't a code patch; it was a topology change: give each source session its own branch so a variant only ever sees one session's universe. The symptom only appeared under concurrency, which is the tell of a structural problem rather than a logic one. (I'd hit the same shape of bug before — a merge that should have been scoped was overwriting shared state — and the lesson is the same: isolate the namespace, don't try to out-clever the leak.)</p>

<h2>Pattern 2: blobstore-native metadata as a queryable sidecar</h2>
<p>Sessions needed a few dozen queryable properties for faceted search in the UI. The default 2026 answer is "stand up a Postgres table." I didn't.</p>
<p>For each uploaded session, the tagger writes a single 0-byte marker object and attaches the tags to it as <strong>native object metadata</strong> (GCS Object Contexts — S3 has analogues). The faceted-search API reads markers directly from the bucket:</p>
<pre><code class="hljs language-python"><span class="hljs-comment"># 0-byte marker, idempotent</span>
blob.upload_from_string(<span class="hljs-string">b""</span>, content_type=<span class="hljs-string">"application/octet-stream"</span>)

<span class="hljs-comment"># attach tags as the object's native metadata</span>
patch_body = {<span class="hljs-string">"contexts"</span>: {<span class="hljs-string">"custom"</span>: tags_payload}}
requests.patch(metadata_api_url, headers=auth, json=patch_body)
</code></pre>
<p>Tags are extracted from a handful of source files in the session (control records, device manifests, session-info JSON, end-of-session markers) plus a directory listing, then a derived-tags pass adds the values the UI actually filters on — duration buckets, chunk-count buckets, an estimated size, a simulated-vs-real flag, and a scrubbed flag set once catalog ingestion has run.</p>
<p>Why blobstore metadata instead of a table:</p>
<ul>
<li><strong>Colocated with the data.</strong> Tag and object live in the same namespace. A session that moves or gets copied keeps its tags. There is no "table out of sync with the bucket" failure mode, because there is no table.</li>
<li><strong>Server-side queryable.</strong> The object-storage API filters on this metadata; you don't fetch every row and filter in Python.</li>
<li><strong>No schema migrations.</strong> Adding a tag is editing the extractor and re-running the tagger over the corpus. No <code>ALTER TABLE</code>, no rollback plan. We added six tags over the system's life and not one needed an API deploy.</li>
<li><strong>Permissions piggyback on bucket IAM.</strong> Whoever can see the session can see its tags.</li>
</ul>
<p>The cost is real: no JOINs, no transactional multi-object updates, and aggregations are slower than a Postgres index. We bought back the read latency with a background cache (below). The endpoint joins two sources of truth in memory — object metadata answers <em>what we have</em>, the lake answers <em>what's imported</em>. If I ever needed cross-session aggregates, I'd add a read-only Postgres mirror populated from the tagger, not move the tags off the blobstore.</p>
<h2>Pattern 3: cheap rewrite, deterministic adversarial validator</h2>
<p>The replacement itself is deliberately dumb — four sequential string replacements over the four identifiers (a session UUID, a site name, a job GUID, a device name), no regex, no JSON-aware traversal:</p>
<pre><code class="hljs language-python"><span class="hljs-keyword">def</span> <span class="hljs-title function_">replace_all</span>(<span class="hljs-params">self, content: <span class="hljs-built_in">str</span></span>) -> <span class="hljs-built_in">str</span>:
    out = content.replace(<span class="hljs-variable language_">self</span>.old_session_id, <span class="hljs-variable language_">self</span>.new_session_id)
    out = out.replace(<span class="hljs-variable language_">self</span>.old_site,       <span class="hljs-variable language_">self</span>.new_site)
    out = out.replace(<span class="hljs-variable language_">self</span>.old_job_guid,   <span class="hljs-variable language_">self</span>.new_job_guid)
    out = out.replace(<span class="hljs-variable language_">self</span>.old_device,     <span class="hljs-variable language_">self</span>.new_device)
    <span class="hljs-keyword">return</span> out
</code></pre>
<p>What makes a dumb rewriter safe is a deterministic verifier behind it. Before any variant is allowed to merge, a scanner walks every object on the branch and fails the merge if any original identifier survives — in a path or in metadata-file content:</p>
<pre><code class="hljs language-python"><span class="hljs-keyword">for</span> obj <span class="hljs-keyword">in</span> all_objects:
    <span class="hljs-keyword">if</span> <span class="hljs-built_in">any</span>(ident <span class="hljs-keyword">in</span> obj.path <span class="hljs-keyword">for</span> ident <span class="hljs-keyword">in</span> originals):
        violations.append((obj.path, <span class="hljs-string">"path"</span>, ident))

metadata_files = [o <span class="hljs-keyword">for</span> o <span class="hljs-keyword">in</span> all_objects <span class="hljs-keyword">if</span> o.path.endswith(METADATA_EXTS)]
<span class="hljs-keyword">with</span> ThreadPoolExecutor(max_workers=scan_workers) <span class="hljs-keyword">as</span> pool:
    <span class="hljs-keyword">for</span> f <span class="hljs-keyword">in</span> as_completed(pool.submit(_check_content, o, originals)
                          <span class="hljs-keyword">for</span> o <span class="hljs-keyword">in</span> metadata_files):
        violations.extend(f.result())
</code></pre>
<p>Both passes are linear; binary recording chunks are skipped because they're identifier-free by construction. The generator is allowed to be dumb precisely because the gate is not — the same shape as pairing a hallucination-prone generator with a deterministic check at the boundary. A real leak (we tested by deliberately breaking the replacement) shows up as a violation tuple naming the file, the field, and the offending identifier.</p>
<p><strong>The merge-serialization gotcha.</strong> With the gen and validate tiers both scaled out, validate-workers were calling <code>merge</code> concurrently against the same run repo, and the merges hit object-storage <strong>429 rate-limit</strong> errors from concurrent SSTable writes in the lake's metadata layer. The fix wasn't more retries; it was a Redis lock that serializes merges per repo, with exponential backoff for transient conflicts:</p>
<pre><code class="hljs language-python"><span class="hljs-keyword">def</span> <span class="hljs-title function_">_acquire_merge_lock</span>(<span class="hljs-params">redis, repo_id</span>):
    deadline = time.time() + LOCK_ACQUIRE_TIMEOUT
    key = <span class="hljs-string">f"merge-lock-<span class="hljs-subst">{repo_id}</span>"</span>
    <span class="hljs-keyword">while</span> time.time() &#x3C; deadline:
        <span class="hljs-keyword">if</span> redis.<span class="hljs-built_in">set</span>(key, worker_id, nx=<span class="hljs-literal">True</span>, ex=MERGE_LOCK_TTL_SECS):
            <span class="hljs-keyword">return</span> <span class="hljs-literal">True</span>
        time.sleep(LOCK_POLL_INTERVAL)
    <span class="hljs-keyword">return</span> <span class="hljs-literal">False</span>
</code></pre>
<p>The lock holds for the merge itself — typically under a few seconds — with a TTL as the safety net for a worker that dies mid-merge. After this, a multi-worker run reports zero retries at the merge step; before, the same workload surfaced a 429 within the first handful of merges. The honest caveat: serializing per repo caps merge throughput at one at a time. It's fine because per-merge wall-clock is small and the workload is bursty and gen-bound. At an order of magnitude more variants it would need lake-side merge batching or repo sharding instead.</p>
<h2>Pattern 4: database-internal hooks as the output channel</h2>
<p>When a variant passes validation and merges to <code>main</code>, the thing that announces it isn't a worker, a watcher, or a polling export service — it's a script running <em>inside</em> the lake, fired on merge, that lists every object on the commit and publishes one message per file:</p>
<pre><code class="hljs language-lua"><span class="hljs-comment">-- enumerate every object on this commit (paginated)</span>
<span class="hljs-keyword">repeat</span>
  <span class="hljs-keyword">local</span> page = lakefs.list_objects(repo, ref, prefix, marker, page_size)
  <span class="hljs-keyword">for</span> _, obj <span class="hljs-keyword">in</span> <span class="hljs-built_in">ipairs</span>(page.results) <span class="hljs-keyword">do</span>
    batch[#batch+<span class="hljs-number">1</span>] = build_payload({
      logical_path  = obj.<span class="hljs-built_in">path</span>,             <span class="hljs-comment">-- where it should land</span>
      physical_addr = obj.physical_address, <span class="hljs-comment">-- where the bytes live now</span>
      commit_id = ref, run_id = run_id,
    })
    <span class="hljs-keyword">if</span> #batch >= <span class="hljs-number">100</span> <span class="hljs-keyword">then</span> <span class="hljs-built_in">flush</span>(batch); batch = {} <span class="hljs-keyword">end</span>

<p>The consumer reads the physical address and does a <strong>server-side copy</strong> from the lake's blockstore straight to the destination bucket:</p>
<pre><code class="hljs language-python">gcs.copy(src=physical_address, dst=<span class="hljs-string">f"gs://<span class="hljs-subst">{logical_bucket}</span>/<span class="hljs-subst">{object_path}</span>"</span>)
</code></pre>
<p>Server-side copy charges storage, not egress — so a multi-gigabyte variant lands in the destination bucket for the cost of metadata I/O. No bytes traverse the network; no worker pod ever has to hold them.</p>
<p>Why doing this inside the database earns its keep:</p>
<ul>
<li><strong>Atomicity.</strong> The hook runs after the merge, in the same transaction. A variant cannot be "merged but unannounced" or "announced but unmerged" — either the hook runs and the commit sticks, or it fails and the merge rolls back. That's a stronger guarantee than any commit-polling architecture I've built.</li>
<li><strong>One message per file.</strong> An earlier version published a single "variant ready" message with a manifest URL; the consumer then re-listed the variant to find files. That was racy under multi-session runs and gave no per-file backpressure. Per-file messages push fan-out into well-understood queue machinery and let the consumer parallelize naturally.</li>
<li><strong>The consumer depends on the queue and the bucket, nothing else.</strong> The physical address is already in the message, so the CI side can run even if the lake is down.</li>
</ul>
<p>The cost: the Lua sandbox is restricted — no arbitrary network libraries, no schema validation, hand-rolled JSON and base64 marshalling, and debugging means reading pod logs because there's no REPL against a live repo. The trade was acceptable: a couple hundred lines of Lua replaced a Go service with its own deployment, identity, and retry semantics. The one warning for anyone replicating it: post-merge hooks fire per-merge, not per-commit. Publishing <em>all</em> files (rather than the commit delta) was deliberate — the downstream CI is stateless about prior runs, so each variant should arrive as a self-contained event.</p>
<h2>Pattern 5: workers that cost nothing at rest</h2>
<p>Every worker is a single-shot Job triggered by Redis-stream lag, with KEDA deciding how many pods should exist:</p>
<pre><code class="hljs language-yaml"><span class="hljs-attr">triggers:</span>
  <span class="hljs-bullet">-</span> <span class="hljs-attr">type:</span> <span class="hljs-string">redis-streams</span>
    <span class="hljs-attr">metadata:</span>
      <span class="hljs-attr">stream:</span> <span class="hljs-string">testgen-queue</span>
      <span class="hljs-attr">consumerGroup:</span> <span class="hljs-string">gen</span>
      <span class="hljs-attr">lagCount:</span> <span class="hljs-string">"5"</span>
</code></pre>
<p>Pods come up, claim entries with <code>XREADGROUP</code>, process, <code>XACK</code>, and exit. Because they're Jobs rather than long-lived consumers, a crash is recoverable — an un-ACKed entry rolls back to the stream and the next pod picks it up. <code>minReplicas=0</code> is the whole point: idle cost is zero, the price is a cold start before the first variant, which for batch test-data generation is fine.</p>
<p>Two scars from making this real. First, the platform's queue scaler needed an IAM grant that was blocked, so a tiny always-on relay pod subscribes to the upstream topic and <code>XADD</code>s into Redis — an extra hop and an extra place to lose a message, but it shipped and reused the stream-trigger config the other workers already had. Second, the Redis instance was shared with another team's job runner using the default DB and <code>allkeys-lru</code> eviction; our streams got evicted under memory pressure and poison messages from old runs re-entered our consumer groups after restarts. The fix was a one-line move to an isolated, unused Redis DB index — found only by attaching an ephemeral pod and watching <code>MONITOR</code>, because a managed Redis isn't a pod you can port-forward into.</p>
<h2>Pattern 6: a background cache that unblocks the UI</h2>
<p>The catalog page lists every session with tags and import status. The first version did it synchronously — list object-metadata markers, list lake prefixes, join in memory — at roughly 45 seconds per page load. Unusable. A small module prewarms both sides on an interval; the endpoint reads from an in-memory snapshot and returns in about 0.65 seconds, exposing the snapshot age so the UI can show a staleness badge. The point isn't only the speedup: it unblocked a whole class of UI work (filter-driven refetches, a polling audit dashboard) that simply can't run against a multi-second backend. The refresh schedule lives in the cache module, not the endpoint — which is what makes per-request latency deterministic instead of paying the slow work on every call.</p>
<h2>What I'd change</h2>
<ul>
<li><strong>Drop the relay.</strong> With the right IAM, the platform's native queue scaler replaces it. It's one pod that shouldn't have to exist.</li>
<li><strong>Merge serialization is a stopgap.</strong> A single per-repo lock caps throughput at one merge at a time. Past a few hundred variants, I'd batch merges lake-side or shard runs across repos.</li>
<li><strong>The four-string replace is too dumb.</strong> If one identifier is a substring of another, the sequential <code>.replace()</code> chain corrupts the wrong thing. The validator catches it, but it surfaces as "generation always fails for this input" rather than a clean error. A single-pass regex with word boundaries, or JSON-aware traversal, fixes it — the validator stays regardless.</li>
<li><strong>The hook would be better as WASM.</strong> Lua is right for the deployment story (no extra service) and wrong for the developer story (no REPL, hand-rolled JSON). A WASM module from Go or Rust would keep the shape and unlock the tooling, if the lake ever supports it.</li>
<li><strong>Make the cache survive restarts.</strong> Today a pod restart cold-starts the cache and the first request eats the full latency. A Redis-backed cache trades one sub-millisecond hop for restart resilience.</li>
<li><strong>Repo-per-run leaks.</strong> Deleting a run repo leaves orphaned blobstore artifacts. Fine at the scale we tested; a slow leak at production scale. Either a cleanup job that scrubs the namespace after deletion, or one repo with branches-per-run instead of repos-per-run.</li>
</ul>
<h2>What holds up</h2>
<p>Two ideas I'd build the same way tomorrow.</p>
<p><strong>The branch is the variant.</strong> When most of your bytes don't change between derivatives of a dataset, don't copy them. Use a versioned store whose branch primitive is zero-copy, let the branch <em>be</em> the deliverable, and delete the rendering pipeline that materializes what you already have a pointer to. The savings compound at every layer — storage, egress, time-to-first-test, audit footprint.</p>
<p><strong>Put metadata where the data is.</strong> If your metadata describes objects in a bucket, a separate store creates a permanent synchronization problem and a second permissions axis. Native object metadata is queryable enough for faceted search and colocated with what it describes. Reach for a relational store when you have relational data — not when you have annotations on a corpus.</p>
<p>Everything else — the specific lake, KEDA, the Redis lock, the cache, even the four-identifier rewrite — was the right call for the constraints I had and is cheerfully substitutable.
